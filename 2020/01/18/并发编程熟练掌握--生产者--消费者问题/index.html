并发编程熟练掌握–生产者 和 消费者问题

生产者–消费者问题本质上是一个多线程协作的模式。可以将生产者–消费者问题理解成两类线程,一类
线程是生产者线程用来生产数据,另一类线程是消费者线程来消费数据。那么怎么将生产者和消费者进行解耦呢？ 如果我们在生产者和消费者中间放一个”数据存放的仓库”,生产者只需要将生产数据丢到”数据仓库中”,而消费者只需要从”数据仓库”中进行消费数据。这样生产者就可以不用关系消费者的怎么进行消费,消费者也可以不用关系生产者怎么进行生产数据的逻辑。
前面的博客中写道过阻塞队列的特点, 思路同样适用于生产者–消费者问题中:
1.当共享数据区域已满的时候,那么生产者就会被阻塞。
2.当共享数据区域为空的时候,那么消费者就会被阻塞。

我们下面用三种思路先解决和实现生产者和消费者问题:
1.使用Object的wait和notify机制
2.使用Condition的await和singal机制
3.使用阻塞队列

wait/notify 等待通知机制

api

1) wait
该方法用来将当前线程置入休眠状态，直到接到通知或被中断为止。在调用 wait()之前，线程必须要获得该对象的对象监视器锁，即只能在同步方法或同步块中调用 wait()方法。调用 wait()方法之后，当前线程会释放锁。如果调用 wait()方法时，线程并未获取到锁的话，则会抛出 IllegalMonitorStateException异常，这是以个 RuntimeException。如果再次获取到锁的话，当前线程才能从 wait()方法处成功返回。
2) notify
该方法也要在同步方法或同步块中调用，即在调用前，线程也必须要获得该对象的对象级别锁，如果调用 notify()时没有持有适当的锁，也会抛出 IllegalMonitorStateException。
该方法任意从 WAITTING 状态的线程中挑选一个进行通知，使得调用 wait()方法的线程从等待队列移入到同步队列中，等待有机会再一次获取到锁，从而使得调用 wait()方法的线程能够从 wait()方法处退出。调用 notify 后，当前线程不会马上释放该对象锁，要等到程序退出同步块后，当前线程才会释放锁。
3) notifyAll
该方法与 notify ()方法的工作方式相同，重要的一点差异是：
notifyAll 使所有原来在该对象上 wait 的线程统统退出 WAITTING 状态，使得他们全部从等待队列中移入到同步队列中去，等待下一次能够有机会获取到对象监视器锁。

wait/notify 的潜在问题

1) notify提前通知

console result:
notifyThread进入代码块
notifyThread开始notify
notifyThreadrun方法结束
waitThread进入代码块
waitThread开始wait

由上述demo和输出结果可以看出, waitThread并不能结束了。 是因为notifyThread的notify先于waitThread的wait方法。那么解决的思路是加一个状态标志,让waitThread在执行wait之前先判断这个状态标志位是否被notifyThread所修改。也就是在notifyThread进行notify后要改变这个状态标志位的值 这样做的话,如果notify先于wait方法,那么wait方法就不会执行了。

console result:
notifyThread进入代码块
notifyThread开始notify
notifyThreadrun方法结束

所以在线程的等待/通知机制中,需要加入一个状态变量, 在notify方法后马上改变该状态值。且在wait方法执行前判断该变量, 让wait方法可以退出循环。一般的做法都需要在wait方法外层包一层while方法。

2)等待wait条件发生变化
如果线程在等待的时候接到通知, 但是在等待方法之后状态发生了变化,如果不进行再次判断的话就可能会出错。

console result:
Productor-Thread 开始添加元素
Consumer-Thread-2 wait方法执行完毕
Consumer-Thread-2 consumer element:Productor-Thread
Consumer-Thread-1 wait方法执行完毕
Exception in thread “Consumer-Thread-1 “ java.lang.IndexOutOfBoundsException: Index: 0, Size: 0
at java.util.ArrayList.rangeCheck(ArrayList.java:657)
at java.util.ArrayList.remove(ArrayList.java:496)
at com.ahang.promotion.concurrent.providerconsumer.ConditionChange$Consumer.run(ConditionChange.java:62)
at java.lang.Thread.run(Thread.java:748)

Process finished with exit code 0

出现异常原因分析：先是consumer1获取了锁,发现lock为空,则进入代码块并执行了wait方法,这时候释放了锁。同样consumer2进行了同样的步骤然后释放了锁。 最后由provider线程获取到锁往lockObject中添加了一个元素,并执行notifyAll将 2个consumer先唤醒。此时两个消费者都会执行remove操作,此时就会抛异常。
解决方法：在上述消费者线程的判断中将if改成while,进行再次判断即可。

3)假死状态
现象：如果是多消费者和多生产者情况，如果使用 notify 方法可能会出现“假死”的情况，即唤醒的是同类线程。
原因分析：假设当前多个生产者线程会调用 wait 方法阻塞等待，当其中的生产者线程获取到对象锁之后使用 notify 通知处于 WAITTING 状态的线程，如果唤醒的仍然是生产者线程，就会造成所有的生产者线程都处于等待状态。
解决办法：将 notify 方法替换成 notifyAll 方法，如果使用的是 lock 的话，就将 signal 方法替换成 signalAll 方法。

总结
1.使用while进行对状态标志的判断。这样可以解决早通知和等待时候状态发生变化的问题。
2.防止假死状态,使用notifyAll而不是notify。
