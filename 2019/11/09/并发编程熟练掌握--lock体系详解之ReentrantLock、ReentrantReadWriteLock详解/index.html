并发编程熟练掌握–Lock体系详解之ReentrantLock、ReentrantReadWriteLock锁详解

ReentrantLock介绍

ReentrantLock是实现了Lock接口，支持重入性。重入性指对共享资源能够重复加锁，即当前线程获取该锁再次获取不会被阻塞。synchronized通过隐式的方式支持了锁的重入，实现
的方法是通过获取自增,释放自减方式来实现重入。而通过上接我们看过ReentrantLock的源码发现，ReentrantLock支持公平锁和非公平锁。那么我们就来分析重入性原理和公平锁、非公平锁。

重入性的实现原理

重入性是指共享资源能够重复加锁,获取到锁的线程再次获取该锁不会被阻塞。那么这句话中就存在两个思考：
1.线程获取锁的时候，如果已经获取锁的线程是当前线程则再次获取锁成功。
2.因为获取了N次锁，那么只有在锁统一被释放了N次后该锁才算被真正完全释放了。

公平锁和非公平锁

ReentrantLock的源码可以看出有两种锁的实现：公平锁和非公平锁。 其公平性体现在时序上，即如果一个锁是公平的，那么锁的获取顺序应该严格按照请求上的时间顺序，满足FIFO

通过源码可以看出,公平锁相对于非公平锁仅仅只是多一个hasQueuedPredecessors方法的判断，方法名就可以看出判断当前节点是否有前驱节点,如果有前驱节点那么说明有线程比当前节点更前进来，根据公平性的定义那么后进来的线程就会获取锁失败。如果当前节点没有前驱节点的话，才会去做后面的逻辑。公平锁每次都是从同步队列中的第一个节点获取到锁，而非公平锁则不一定，有可能刚释放了就又获取到了该锁。

公平锁 VS 非公平锁
1.公平锁每次获取到锁为同步队列中的第一个节点, 保证了请求资源时间上的绝对顺序。而非公平锁有可能刚释放锁的线程下次继续获取到该锁 ，则有可能导致有的线程永远获取不到锁，造成”饥饿现象”
2.公平锁为了保证绝对的时序性，需要频繁的上下文切换，而非公平锁会降低一定的上下文切换，从而减少性能。 如上的无参构造函数就可以看出ReentrantLock默认选择的是公平锁，从而保证了系统更大的吞吐量

什么是读写锁

通常为了保证线程安全，我们场使用的就是独占锁。如 synchronized 或者 Lock包中的实现了Lock接口的ReentrantLock。而独占所我们已经很清楚了, 任一时刻只有一个线程可以获取到锁的状态，而其他线程就会被阻塞。但是实际开发中, 我们很有可能会遇到这种场景, 需要并发处理的业务逻辑中，读数据很多，写数据很少。那么如果仅仅是读数据并不会影响数据的正确性(出现脏读),而如果再这种业务场景中, 依然使用独占锁的话那么服务端的性能瓶颈就会很低。
这时候就出现了一种新的处理方式, 读写锁。 适用于读多写少的业务场景，ReentrantReadWriteLock锁允许用一个时刻被多个读线程访问，但是在写线程访问的时候，所有的读线程都被阻塞。

写锁的获取和释放

同步组件的实现聚合了同步器AQS, 并通过重新AQS的同步组件来实现同步语义。同样，写锁也依靠这种方式来实现的。同一时刻允许多个读线程访问，而在写线程访问时候所有的读写线程都被阻塞。那么可以知道写锁是独占锁。

其中static final int EXCLUSIVE_MASK = (1 << SHARED_SHIFT) - 1; 位运算表示为1左移16位 然后 -1，然后与 c做 与 操作, 这样得出的结果就是同步状态的低16位(对这里&操作有疑惑,推荐自己手写一遍位运算就清楚了)。 而exclusiveCount方法是获取写锁的获取次数的，也就表示同步状态的低16位表示写锁的获取次数。 同理可以去看看读锁，先说结论： 同步状态的高16位表示读锁的获取次数。

我们来小结一下，写锁干了啥. 当读锁以及被读线程获取 或者 写锁以及被其他写线程获取的时候，则写锁获取失败。否则获取成并且支持重入。

读锁的获取

读锁是共享锁，即在同一时刻该锁能被多个线程获取到。那么读锁是怎么实现的呢？ 现在是不是脑子里第一个闪现出来的是AQS, 因为同步组件的实现是通过聚合AQS，并重写AQS的同步组件来实现的同步语义。

通过源码可以知道 当锁被 写线程获取到的时候则读锁获取失败, 否则利用CAS进行更新同步状态。其中源码有个注释，设置重入的同步状态的更新值为 (c + SHARED_UNIT) , 而 SHARED_UNIT = (1 << SHARED_SHIFT) 。 也就是同步状态为 c + 高十六位的值, 也就证明了上面所述同步状态的高十六位表示获取读锁的次数。
